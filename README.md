# 구현 중 이슈
## Jwt (refresh + access)에 관한 고민
### 액세스 토큰만을 사용할 경우
- 액세스 토큰은 stateless이기 때문에 탈취가 되면 서버 입장으로서 관리할 수 있는 방법이 없음 -> 액세스 토큰의 유효기간을 짧게 설정 -> 유저 입장에서는 토큰이 만료될 때마다 재 로그인하여 갱신해줘야 하는 번거로움 발생

- 유효 기간을 길게 가져가면 탈취 당했을 경우 유효 기간이 만료될 때까지 탈취한 공격자가 제약없이 사용할 수 있음

- DB에 유효 액세스 토큰을 저장 -> Jwt는 stateless로 정의 되었지만 DB에 저장하는 순간 그 목적이 깨지게 됨.

### Refresh 토큰 도입
- 리프레시 토큰은 액세스 토큰을 재 발급해주는 역할을 함.
- 리프레시 토큰은 액세스 토큰보다 비교적 긴 유효기간을 가짐, 액세스 토큰이 만료되더라도 비교적 긴 유효기간을 가진 리프레시 토큰을 통해 유저는 로그인 없이 액세스 토큰을 재발급 받을 수 있음.
- 비교적 짧은 액세스 토큰을 탈취 당하더라도, 유효 기간이 짧아서 비교적 괜찮음.

- 하지만 리프레시 토큰을 탈취 당하면 공격자는 무한하게 액세스 토큰을 발행할 수 있음.

- 리프레시 토큰을 DB에 저장하여 유저당 유효한 리프레시 토큰 1개 만을 사용할 수 있도록 구현함, 로그인 또는 액세스 토큰이 만료되어 재발급할 때 리프레시 토큰이 업데이트 되도록 구현함.
  - 리프레시 토큰은 관계형 DB에 저장하지 않고, 메모리에 저장하는 Redis를 사용해서 유효기간이 지나면 삭제가 되도록 구현함. **(Redis 도입 이유)**

- 즉, 로그인, 액세스 토큰 재발행할 경우 유저의 고유한 리프레시 토큰이 재생성, 저장됨.

### 도전과제
- 액세스 토큰의 유효 기간은 발급 후 2시간으로 정하고, 리프레시 토큰의 유효 기간은 발급 후 14일로 정하였음.
- 위 기간에 대한 정의 방법도 함께 공부해야할 과제로 남음.

- 한편으로는 유저의 MAC 이나 IP + 타임스탬프를 기반으로 Jwt를 생성하는 방법도 있었지만, Jwt를 택해서 사용하게 됨.
- Jwt보다 더 안전한 인증 및 인가에 대해 공부
- 비밀번호 방식(Know)의 로그인 방식을 사용하고 있지만, 타임스탬프 기반의 인증 코드 생성 방식(Google Authenticator, MS Authenticator 등)의 도입도 고민

## 로그인 및 이메일 인증 여러 번 실패 시 패널티 부여

### 패널티
파이어에 로그인을 여러 번 시도해서 실패하거나, 회원가입 이메일 인증 코드를 여러 번 틀리게 될 경우 해당 이메일의 접근을 10분 간 방지하는 정책을 만들고 싶었다.
하지만 구체적인 방법을 생각해보지는 못 하여, 생각해보고, 구현하게 되었다.

### 패널티 부여 조건
5분 이내에 5번 이상 로그인에 실패하거나, 이메일 인증에 실패할 경우 패널티가 부여됨.

### 조건 확인
로그인 또는 이메일 인증 비즈니스 로직에서 실패했을 경우에 Redis에 (실패한 이메일+'특정 구문'을 키), (이메일+'특정 구문'의 키가 갖는 값+1)을 저장하도록 하였다.
데이터는 5분 동안 메모리에 저장되기 때문에 이메일+'특정 구문'의 값이 5 이상이 되면 패널티 부여 조건이 성립되는 것이다.

### 패널티 부여
패널티 부여 조건이 확인되면 Redis에 10분 간 유효하는 데이터를 생성하여 해당 이메일을 통한 로그인 및 이메일 인증을 못하도록 구현하였다.

### 도전과제
- 로그인 여러 번 실패의 경우 많은 실제 서비스들에서 볼 수 있는데, 내가 사용한 방법과 유사하게 동작하는지, 아니면 완전히 다른 방식으로 동작하는지에 대한 공부가 필요함.
- Redis 말고 다른 방식으로 구현해볼 것.

## Record vs Class
처음 백엔드를 공부할 당시 DTO에 대해서 깊게 고민한 적이 없었고, 비교적 로직이 없을 정도의 코드이기 때문에 검색을 통해 알게된 방식인 Class를 사용했었다.
@Getter나 @Data 어노테이션을 사용하여 DTO를 사용했었는데, record를 알게 되었다.

DTO로만 사용할 목적으로 class 대신 record를 DTO에 사용하게 되었다.
Lombok 어노테이션 선언을 하지 않아도 필요한 기능을 사용할 수 있다는 점, private final로 멤버 변수가 선언된다는 점에서 정하게 되었다. 
### Record
- 기본적으로 멤버 변수가 private final로 생성된다.
- 필드별로 자동으로 Getter를 생성한다.
- 모든 멤버 변수를 인수로 가지는 기본 생성자가 자동으로 생성된다.
